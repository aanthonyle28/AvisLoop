## Objective

Create the server-side data layer for additional business creation: an insert-only server action that creates a new business row (never upsert), scoped save actions for wizard steps that accept explicit `businessId` (avoiding `getActiveBusiness()`), and modify the `/onboarding` page to support `?mode=new` query param so it renders the new-business wizard instead of redirecting to dashboard.

This plan is the data foundation for Phase 56. The UI (Plan 56-02) depends on these actions existing.

## Context

@lib/actions/active-business.ts — `switchBusiness()` we will call at completion

@.planning/phases/56-additional-business-creation/56-RESEARCH.md — Full research with pitfall analysis

## Critical Safety Rules

1. **NEVER use `.upsert()`** anywhere in this plan. Every database write is `.insert()` or `.update()` with explicit `.eq('id', businessId)`.
2. **NEVER call `getActiveBusiness()`** in any new action. The whole point is explicit businessId scoping.
3. **NEVER modify existing `saveBusinessBasics()`, `saveServicesOffered()`, `acknowledgeSMSConsent()`, or `markOnboardingComplete()`** — they are correct for the first-business path.
4. **All `.update()` calls must include `.eq('user_id', user.id)`** as an ownership guard in addition to RLS.

<steps>
1. Create `lib/actions/create-additional-business.ts` with `'use server'` directive

2. Implement `createAdditionalBusiness(input: BusinessBasicsInput)`:
   - Validate with `businessBasicsSchema.safeParse(input)`
   - Get authenticated user via `supabase.auth.getUser()`
   - Call `.insert({ user_id: user.id, name, phone, google_review_link })` — PURE INSERT, never upsert
   - Chain `.select('id').single()` to get the new business ID
   - Call `revalidatePath('/businesses')`
   - Return `{ success: true, businessId: data.id }` or `{ success: false, error }`

3. Implement `saveNewBusinessServices(businessId: string, input: ServicesOfferedInput)`:
   - Validate businessId is truthy
   - Validate with `servicesOfferedSchema.safeParse(input)`
   - Authenticate user
   - Build `timingMap` from `DEFAULT_TIMING_HOURS` for each selected service type
   - Call `.update({ service_types_enabled, service_type_timing, custom_service_names }).eq('id', businessId).eq('user_id', user.id)`
   - Return success/error

4. Implement `createNewBusinessCampaign(businessId: string, presetId: string)`:
   - Validate both IDs are truthy
   - Authenticate user
   - Fetch source preset: `.from('campaigns').select('*, campaign_touches(*)').eq('id', presetId).eq('is_preset', true).single()`
   - Insert new campaign: `.from('campaigns').insert({ business_id: businessId, name: source.name, service_type: source.service_type, status: 'active', is_preset: false, personalization_enabled: source.personalization_enabled ?? true }).select('id').single()`
   - Insert touches if source has any: map each touch to `{ campaign_id: newCampaign.id, touch_number, channel, delay_hours, template_id }`
   - Return success/error

5. Implement `completeNewBusinessOnboarding(businessId: string)`:
   - Validate businessId
   - Authenticate user
   - Call `.update({ sms_consent_acknowledged: true, sms_consent_acknowledged_at: new Date().toISOString(), onboarding_completed_at: new Date().toISOString() }).eq('id', businessId).eq('user_id', user.id)`
   - Return success/error

All four functions follow the same pattern: `'use server'`, validate input, authenticate, operate on explicit businessId with `.eq('user_id', user.id)` ownership guard.
</steps>
</task>
